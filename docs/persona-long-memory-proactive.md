# 人物设定与主动消息

## 回复开头括号

服务器会在网页聊天和微信回复返回前清理开头的动作/旁白括号，例如：

- `（她低头笑了笑）我在。`
- `(轻声) 我在。`
- `【沉默片刻】我在。`

清理后会只发送正文：`我在。`。系统提示词也会要求模型不要把动作旁白放在回复开头。

## 长篇人物设定

人物编辑页里有「长篇人物设定 / 小说资料」字段，适合粘贴从小说中整理出来的目标人物背景、经历、关系、说话习惯和关键剧情。每次对话构造提示词时，后端会把这个字段作为独立的长篇背景注入，最多使用前 32,000 个字符。

建议只保留目标人物相关内容：

- 人物经历、身份、关系网。
- 关键剧情节点和心理变化。
- 口头禅、语气、称呼习惯。
- 绝对不能违背的设定。

如果原文很长，优先整理成「人物专属资料」再粘贴进这个字段；不要把整本小说无筛选地塞进去。

## 对话记忆

用户中心的「AI 设置」里，「上下文消息数」控制每次对话会回看最近多少条聊天记录，当前上限是 100 条。这个设置解决的是近期对话记忆；人物背景和性格应写在人物编辑页的长篇设定里。

社交端和网页端还会额外注入「对话连续性」提示：模型会看到最近对话时间线、上一条用户消息、上一条角色回复，以及是否存在角色刚问过但用户还没有回应的问题。这样主动消息不会把每个时间点当作全新开场，例如 18:00 刚问过吃饭而用户没回，20:30 更应该自然跟进“刚才问你吃饭，你也没回我”，而不是重新问一个相似问题。

QQ/微信文本入口有短时间合并队列：连续发来的短消息会先合并成一段连续话语再交给模型。如果模型思考期间用户又发来新消息，旧回复会被判定为过期，不发送也不写入角色回复记录；下一轮会把上一条角色回复之后累积的未回应用户消息当作一个整体来理解，只生成一次综合回复，避免突然把旧消息逐条补答出来。

## 长期记忆整理

长期记忆使用 `memories` 表保存，和最近聊天上下文分开。它现在按“记忆卡片”保存，不再只是标题和描述。

每张记忆卡片都有这些字段：

- `source`：记忆来源。`manual` 是手动添加，`chat` 是从近期聊天提取，`daily_summary` 是每日整理，`source_material` 是小说/原著资料。
- `memoryType`：记忆类型。常用类型有 `user_fact` 用户事实、`preference` 偏好习惯、`promise` 承诺约定、`relationship_event` 关系事件、`emotional_moment` 情绪节点、`conflict` 冲突修复、`open_loop` 未完成话题、`source_fact` 原著资料。
- `importance`：重要程度，1-5。越高越容易被召回。
- `confidence`：可信程度，1-5。越低越说明它可能只是推测，召回时会被降权。
- `keywords`：召回关键词，用于后续匹配。
- `emotion`：这条记忆的情绪色彩，例如委屈、开心、心软、低落。
- `evidenceMessageIds`：这条记忆来自哪些原始消息，避免模型凭空把猜测写成事实。
- `status`：`active` 会参与召回，`archived` 和 `contradicted` 默认不参与召回。

服务启动后会自动开启每日记忆整理任务，默认在本地时间 03:20 之后整理前一天聊天，并向前补整理最近 3 天缺失的记忆。每日整理会为当天提取若干张结构化记忆卡片；同一天已经有 `daily_summary` 来源的记忆后不会重复整理。

每日整理只保留以后对话真正需要记住的信息，例如：用户稳定事实、偏好、承诺、关系进展、未解决问题、重要情绪节点。普通寒暄、重复催睡和无信息量闲聊会被跳过。可用环境变量调整：

- `DAILY_MEMORY_ENABLED=false`：关闭自动整理。
- `DAILY_MEMORY_HOUR=3` / `DAILY_MEMORY_MINUTE=20`：调整每天整理时间。
- `DAILY_MEMORY_CATCH_UP_DAYS=3`：服务启动后向前补整理的天数。
- `DAILY_MEMORY_MIN_MESSAGES=4`：当天少于多少条消息时不整理。
- `DAILY_MEMORY_MAX_CHARS=8000`：单日整理传给模型的最大聊天字符数。

正常聊天时不会把所有长期记忆都塞给模型。只有用户问到“记得、之前、上次、承诺、关系、爱意”等，或本轮被识别为关系/情感表达场景，或隐藏思考层认为需要查记忆时，后端才会从长期记忆里选少量相关条目注入提示词，减少 token 消耗，同时保留必要的连续性。

## 回复前的回合结构

一次角色回复现在分成几层：

1. 人物画像：角色是谁，性格、关系、说话方式、原著背景是什么。主要来自 `personaData.profileSections`。
2. 当前时间：真实世界现在几点、星期几。它只是时间事实。
3. 生活状态：角色此刻大概在做什么，例如睡觉、上班、通勤、晚上在家。它由当前时间推导，也可能被短期运行状态覆盖。
4. 情绪状态：角色面对这段关系时的心情倾向，例如温柔、俏皮、思念、低落、开心、疏离。它不是生活状态；一个人可以“晚上在家”但心情低落，也可以“工作间隙”但语气温柔。
5. 回合规划器：代码里的 `planner`。它不是 AI 人格，而是一个规则判断模块，先粗略判断本轮是什么类型、要查哪类记忆、回复该短还是长，以及有哪些风险。
6. 隐藏思考层：代码里的 `reflection`。这是一次小的 LLM 内部判断，用来让角色先“想一下”。它只输出 JSON，不直接发给用户。
7. 记忆召回：根据用户这句话、planner 判断和 reflection 给出的记忆查询词，从长期记忆里选少量相关卡片。
8. 正式回复：把人物画像、当前时间、生活状态、情绪状态、planner、reflection、召回记忆和最近聊天一起交给模型，生成真正发给用户的话。

`intent` 是本轮意图，例如：

- `daily_chat`：普通日常聊天。
- `affection_expression`：用户在要求爱意、表白、想念或内心话。
- `emotional_support`：用户难过、委屈、累，需要被接住。
- `source_recall`：用户问原著、小说或过去具体情节。
- `correction`：用户在纠正角色记错或乱说。

`risk` 是本轮容易出错的地方，不是危险事件。常见风险：

- `repetition`：重复催睡、重复问吃饭、重复关心。
- `over_reply`：用户只是短句或测试，却回复太长太重。
- `emotion_mismatch`：用户在认真撒娇或难过，角色却敷衍；或者用户只是轻松聊天，角色却过度沉重。
- `persona_drift`：为了更深情而偏离人物性格、语气或价值观。
- `memory_contamination`：把小说资料、模型猜测、玩笑或低可信记忆当成真实共同经历。
- `source_hallucination`：没有原著证据却编剧情。
- `sleep_state_conflict`：生活状态显示角色在睡觉，但回复像完全清醒。

隐藏思考层 JSON 的主要字段：

- `intent`：对本轮意图的细化判断。
- `shouldRecallMemory`：是否需要查长期记忆。
- `memoryQueries`：查记忆时用的短关键词，不是“记忆合唱”，而是检索词。
- `shouldRecordMemory`：这轮是否可能值得以后记住。它只是候选标记，不等于马上写入记忆。
- `innerReaction`：角色内心怎么理解这句话。这借鉴 Hanako 的“内心/心情”思路，但在 Mirrai 里默认只作为隐藏上下文，不直接展示给用户。
- `replyStrategy`：真正回复时应该怎么处理，例如先接住委屈、不要催睡、短一点、具体说想念。
- `avoid`：本轮要避免的具体话术或方向。
- `mood`：比 `emotionalState` 更细的一次性心情词。

## 记忆管理界面

聊天页右上角的记忆入口现在会打开「记忆管理」抽屉。它不是只读时间线，可以直接校正长期记忆：

- 按状态筛选：`active` 有效、`archived` 归档、`contradicted` 标错。
- 按类型、来源和关键词筛选。
- 编辑标题、描述、记忆类型、来源、情绪色彩、重要度、可信度和关键词。
- 把暂时不用的记忆归档。
- 把错误记忆标错；标错后可信度会降为 1，并退出正常召回。
- 恢复归档或标错的记忆。

`archived` 和 `contradicted` 不会进入正常长期记忆召回。这样可以在不永久删除历史记录的前提下控制记忆污染。

## 人物状态面板

聊天页顶部的情绪按钮现在会打开「人物状态面板」。它用于观察当前角色为什么会形成某种回复，不会改变聊天正文。

面板分为几组：

- 情绪状态：当前 `emotionalState`，也可以手动切换。
- 生活状态：由当前时间推导出的日程时段、状态类型和回复可用性；如果被叫醒，还会显示短期 `drowsy_awake`。
- 记忆概览：有效记忆数、高重要记忆数、低可信记忆数、原著资料块数量，以及最近被召回的记忆。
- 最近隐藏思考：最近一轮的 `intent`、记忆模式、`innerReaction`、`replyStrategy`、记忆查询词、风险和避免事项。
- 人物画像摘要：身份/关系、性格、关系记忆、说话方式和当前运行模块。

最近隐藏思考只保存在 `personaData.runtimeDiagnostics`。它属于运行诊断，不属于长期记忆，也不会直接注入为人物事实。

## 自动记忆沉淀与治理

聊天正式回复保存之后，会进入 `memory-consolidation`。它属于“回复后处理”，不影响本轮已经发出的消息。

自动沉淀默认很保守：

- 用户消息过短、测试、普通寒暄会跳过。
- 本轮主要依赖原著/资料召回时会跳过，避免把小说资料误写成共同经历。
- 只有隐藏思考层标记 `shouldRecordMemory=true`，或本轮包含明确的偏好、承诺、冲突、关系进展、地点/课程/工作等长期信号时才尝试。
- 即使尝试，也只接受 `importance >= 4` 且 `confidence >= 3` 的记忆；承诺、偏好、冲突、未完成话题可以稍微放宽到重要度 3。
- 每轮最多写入 2 条记忆卡片。

写入前会进入 `memory-governance`。它属于“记忆治理层”，作用是防污染：

- 如果新卡片和已有有效记忆高度重复，跳过写入。
- 如果新卡片与旧记忆明显冲突，且新卡片更可信/更重要，旧记忆标记为 `contradicted`。
- 如果新卡片关闭了一个旧的 `open_loop`，旧的未完成话题会归档为 `archived`。
- 普通召回只使用 `active` 记忆，归档和标错的记忆保留在管理界面里，但不参与正常回复。

最近一次自动沉淀结果也会显示在人物状态面板的「最近隐藏思考」区域，包括状态、跳过原因、写入数量和候选卡片标题。

## 角色日程

后端内置了王芃泽的默认生活日程和状态指示表，作为第一版真实感约束。每个时间段都有基础状态 ID、状态类型、回复可用性、行为规则和下一步转移提示：

- 工作日：早上去南京研究所，08:00-11:30 上午工作，11:30 午饭，14:00-17:30 下午工作，17:30 后下班回家，晚上在家泡茶、看书、整理资料，23:30 后准备睡觉。
- 周六：家里收拾、看书、买菜或偶尔回所里拿资料，晚上更适合自然聊天。
- 周日：上午在家，下午准备下周研究所材料，晚上收心。
- 睡眠时段：普通 QQ/微信社交消息默认不立即回复；如果用户明显有急事、难受、害怕、明确叫醒或需要陪伴，则允许回复。

当前基础状态包括：

- `sleeping`：睡眠，普通消息静默，急事或叫醒才回复。
- `morning_waking` / `weekend_morning`：刚醒和早晨收拾，能回但语气低缓。
- `commuting_to_work` / `commuting_home`：通勤，适合短促报平安，不展开长篇。
- `working_morning` / `working_afternoon` / `sunday_work_prep`：工作或准备工作，回复应短，像工作间隙抽空看手机。
- `lunch_break` / `dinner_at_home`：吃饭，适合日常关心吃饭、下课、晚饭等。
- `midday_rest` / `weekend_errands`：休息或外出琐事，可以短聊但带当下生活感。
- `evening_home`：晚间家中，最适合自然聊天和语音。
- `night_reading`：夜里看书或收心，语气更安静、更近，但不要每次催睡。
- `pre_sleep`：准备睡觉，短句、低声、准备收尾。

这个日程会注入人物系统提示词，所以普通回复、图片/表情包回复、固定时间主动消息、随机日常存在感消息都会知道当前大致状态。它不会改数据库结构；以后如果需要在网页里编辑一周七天的自定义日程，可以再把这套默认日程升级为 `personaData` 配置。

睡眠时段还有一个短期运行状态：如果用户说“醒醒”“你醒了吗”“别睡”或明显有急事，后端会在 `personaData.runtimeLifeState` 里写入 `drowsy_awake`，默认持续 20 分钟。这个状态下后续普通消息会继续得到回复，但提示词会要求角色保持半睡半醒、迷迷糊糊、低声短句的状态；20 分钟内没有继续聊天则自动回到日程里的睡眠状态。

## 主动消息

人物编辑页里可以开启「主动消息」，添加多个 `HH:mm` 时间点。运行时调度器每分钟检查一次：

- 只有人物状态为 ready 时才会发送。
- 优先发送到该人物已绑定的 QQ 联系人；如果没有 QQ 绑定，才回退到微信绑定联系人。
- 有 QQ 绑定时要求 QQ OneBot/NapCat 已连接；没有 QQ 绑定时要求微信机器人已登录。
- 每个时间点每天只发送一次，但不会卡在精确分钟发送。调度器会为每个角色、每个日期、每个设定时间生成一次固定随机落点，范围是设定时间前 10 分钟到后 10 分钟。例如 `12:00` 当天可能实际在 `11:53` 或 `12:08` 发送。
- 随机落点会写入 `personaData.proactiveMessages.randomizedSchedule`，同一天不会每分钟重新随机，避免重复发送或时间漂移。
- 内容由模型随机生成，但会使用人物系统提示词和长篇设定，避免脱离人物背景。
- 生成前会回看最近对话时间线，优先延续未回应的问题、刚发生的聊天状态和当前日程，不应重复刚问过的话或和前文行程冲突。

主动消息配置保存在 `personaData.proactiveMessages`：

```json
{
  "enabled": true,
  "times": ["09:00", "22:30"],
  "stylePrompt": "像平时自然想起我一样发一句，不要解释自己是定时消息。",
  "randomizedSchedule": {
    "windowMinutes": 10,
    "days": {
      "2026-05-08": {
        "09:00": {
          "baseDate": "2026-05-08",
          "baseTime": "09:00",
          "actualDate": "2026-05-08",
          "actualTime": "08:54",
          "offsetMinutes": -6
        }
      }
    }
  },
  "lastSent": {
    "09:00": "2026-05-08"
  }
}
```
