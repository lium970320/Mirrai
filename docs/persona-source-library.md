# 人物原著资料库

Mirrai 支持给单个人物绑定原著/长篇资料库。资料库不是每轮聊天都进入上下文，而是在用户明显询问原著、回忆、旧事、人物经历或特定事件时触发检索。

## 触发规则

普通日常聊天不会查资料库，例如“吃饭了吗”“在干嘛”“想你”。以下类型会查：

- “你还记得……吗”
- “小说里……”
- “当年 / 以前 / 那时候……”
- “老鹰峡 / 北京治疗 / 左臂 / 王小川 / 姚敏 / 林慧珍 / 研究所……”等原著事件或人物词

检索到的片段会以“内部证据”的形式进入提示词。模型被要求不要说“资料库”“检索结果”，而是用人物自己的口吻自然回忆；涉及地点、顺序、动作和人物关系时，必须优先按命中的原文片段回答。

如果命中的片段不足以确认某个具体细节，角色应该自然地说记不准或只说确定的部分，不能用大致剧情、常识或气氛补编情节。

为了降低“先答对一半、后面续编”的问题，原著/回忆类问题会走更严格的生成链路：

- 无命中片段时，也会进入“不要编”的证据模式，而不是退回普通角色聊天。
- 生成时会要求短答，优先只回答用户问到的那一点。
- 生成后会做一次低温证据复核，把原文片段没有支撑的地点、人物、顺序、动作和后续剧情删掉，必要时改成角色自然地承认记不准。

后续纠错句也会触发资料库，例如“你瞎说”“不是这样”“再想想”“当时具体情形不是这样”。这类句子会带上最近的用户原著问题继续检索，但不会把机器人上一轮可能编错的回答拿去当检索依据。

## 重新导入

```powershell
cd F:/Code/Mirrai
node scripts/import-persona-source.mjs --persona-id 1 --file "C:/Users/Lenovo/Downloads/爱人随风而来-完整版_南无.epub" --title "爱人随风而来"
```

导入脚本会替换同一本 epub 的旧索引，不会清空聊天记录。

## 数据库

使用现有 Neon PostgreSQL，不需要第二个数据库。新增表：

- `persona_sources`
- `persona_source_chunks`

第一版是关键词和事件词混合检索，未使用 embedding。检索会先找命中片段，再自动带上前后相邻片段，减少断章取义；展示给模型的是命中词附近的原文，而不是每段固定开头。

当前问题会优先于上一轮话题。比如刚问完老鹰峡，又问“湖里洗澡踢到哪里”，检索会优先用“湖边 / 洗澡 / 踢坏”等当前问题线索，而不是继续沿用老鹰峡的旧线索。

后续如果发现“语义相近但关键词不同”的问题较多，再加向量检索层。
