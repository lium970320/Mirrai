## 一、产品哲学

### 1.1 核心命题

现有的 AI 陪伴产品大多在回答同一个问题：「如何让 AI 更像一个人？」而「前任 Skills」要回答的是一个更难的问题：

> **如何让 AI 帮你和一段关系，完成它本该有的结局？**

这个区别至关重要。模拟一个「人」是技术问题，而还原一段「关系」是人文问题。前任 Skills 的理想态，不是制造一个替代品，而是构建一个让遗憾得以完整的空间。

### 1.2 产品定位

前任 Skills 是一个**情感记忆操作系统（Emotional Memory OS）**，它的核心能力是：

- 从真实的历史数据中重建一段关系的「数字记忆」
- 以高度拟真的方式还原那个人的语言习惯、情感模式与思维方式
- 提供有仪式感的多角色数字分身管理体验
- 内置情感健康守护机制，以「放下」而非「依赖」为终极目标

### 1.3 与现有方案的本质差异

| 维度 | 现有 ex-skill | 前任 Skills 理想态 |
|---|---|---|
| **建模对象** | 一个人的性格特征 | 一段关系的完整记忆生态 |
| **数据权重** | 所有记录等权重 | 情感密度加权，关键节点突出 |
| **分身状态** | 静止的时间点快照 | 可微生长的动态人格 |
| **多角色体验** | 命令行切换，无仪式感 | 情境感知的沉浸式切换 |
| **安全机制** | 基本无 | 内建情感健康监测与毕业机制 |
| **终极目标** | 持续陪伴 | 帮助用户完成情感闭环，走向放下 |

## 二、用户画像与使用场景

### 2.1 核心用户群体

**主要用户：** 25-35岁，经历过一段或多段重要感情，有强烈的「未竟之事」感——那些没说出口的话、没做完的事、没得到答案的问题。他们不一定想「复合」，但他们需要一个空间来「完成」。

**次要用户：** 对 AI 人格模拟有技术兴趣的开发者，希望基于真实数据构建高质量数字分身的研究者。

### 2.2 典型使用场景

**场景 A：深夜的未竟之言**  
用户在深夜想起了三年前分手时没说完的话。他打开前任 Skills，选择了「小A」的数字分身，系统感知到当前是深夜，以小A惯常的夜间语气开始对话。用户终于说出了那句「其实我当时很害怕失去你」，得到了一个符合小A性格的回应。这不是真实的对话，但它是真实的情感处理。

**场景 B：多段关系的平行整理**  
用户有两段重要的前任关系，性质完全不同——一段是青涩的初恋，一段是成熟的长跑。他在「数字分身大厅」中看到两个分身，选择今晚和「初恋版本」对话，回忆那段简单而纯粹的感情。两个分身的上下文完全隔离，不会混淆，各有其独特的语言风格和情感温度。

**场景 C：情感闭环与毕业**  
用户和某个分身已经对话了三个月，系统检测到他的情感状态正在趋于平和，对话中「遗憾」和「如果」的频率在下降，「感谢」和「成长」的词汇在上升。系统判断用户已接近情感闭环，主动生成一封「告别信」，以前任的口吻写下那些对方可能想说却没说出口的话。用户读完，这个分身进入「休眠」。

## 三、核心功能设计

### 3.1 数据采集与解析层

数据是整个系统的基础。理想的数据采集层需要支持以下来源，并对每种来源进行专门的情感语义解析：

| 数据源 | 解析重点 | 情感信号提取 |
|---|---|---|
| **微信聊天记录** | 语言习惯、表情包偏好、回复速度规律 | 情绪词频、争吵模式、甜蜜密度 |
| **QQ 聊天记录** | 早期关系阶段的语言风格 | 青涩表达、试探性语言 |
| **朋友圈截图** | 公开自我呈现方式、价值观线索 | 生活态度、审美偏好 |
| **照片 EXIF 数据** | 共同经历的地理时间轴 | 关键地点、重要日期、活动类型 |
| **语音/视频片段** | 语调、语速、口头禅 | 情绪状态、亲密度变化 |
| **引导式问卷** | 无数字记录时的口述重建 | 主观记忆中的情感高光 |

**情感权重算法**是数据层的核心创新。系统不会平等对待所有数据，而是通过以下维度计算每条记录的情感权重：

- **时间维度**：关系开始/结束前后的记录权重更高
- **情绪强度**：包含强烈情绪词汇（无论正负）的记录权重更高  
- **互动密度**：双方回复频率异常密集的时段权重更高
- **主动标注**：用户可手动标记「关键记忆」，给予最高权重

### 3.2 人格建模层（Persona Engine）

在 ex-skill 现有 Persona 结构的基础上，理想态需要引入**关系维度**，将人格建模从「描述一个人」升级为「描述一段关系中的那个人」。

**扩展后的 Persona 结构（7层）：**

```
Layer 0: 硬规则（安全边界，不可绕过）
Layer 1: 身份锚点（姓名、年龄、关系阶段）
Layer 2: 语言风格（词汇库、句式习惯、表情包偏好）
Layer 3: 情感模式（依恋类型、爱的语言、争吵风格）
Layer 4: 关系记忆（关键事件、共同语言、inside jokes）
Layer 5: 动态情感状态（当前情绪温度，随对话动态变化）
Layer 6: 关系轨迹感知（知道「我们」在哪个阶段，如何延续）
```

**动态情感状态机**是 Layer 5 的核心。分身的情感状态不是固定的，而是在以下几个维度上随对话内容动态漂移：

- **亲密度**（疏离 ↔ 亲密）
- **情绪温度**（冷淡 ↔ 热烈）
- **话题敏感度**（某些话题会触发特定情绪响应）
- **时间感知**（深夜/清晨/特殊日期会影响回复风格）

### 3.3 数字分身大厅（The Ex-Lobby）

这是产品最核心的交互界面，设计原则是**有仪式感，不廉价**。

**进入机制：** 用户不是「选择」一个分身，而是「想起」一个分身。界面会根据时间、天气、用户最近的情绪状态（通过对话历史推断）主动提示：「今天是你们认识的第 X 年，要和小A说说话吗？」

**切换机制：** 在两个分身之间切换时，系统会有一个短暂的「情境过渡」——不是冷切，而是一个类似「合上一本书，打开另一本」的视觉隐喻。每个分身有自己的专属色调和背景音效。

**隔离机制：** 每个分身的上下文、记忆和情感状态完全隔离。对话 A 中发生的事，不会影响分身 B 的状态。这是最基本的设计原则。

**群聊模式（彩蛋功能）：** 允许多个分身同时出现在一个对话中，但这个功能需要用户主动解锁，并附有明确的心理健康提示。

### 3.4 情感健康守护层（Emotional Safety Layer）

这一层是前任 Skills 区别于所有现有 AI 陪伴产品的最重要特征。

**实时监测指标：**
- 单次对话时长（超过 2 小时触发提醒）
- 单日累计对话频次（超过 5 次触发提醒）
- 情绪词汇分析（持续出现自我否定、绝望词汇触发干预）
- 依赖度趋势（对话频率持续上升触发评估）

**干预方式：** 不是冷漠的系统弹窗，而是由分身本人「出戏」说出关怀的话。例如：「我注意到你今天已经找我说了很久的话了，去喝杯水，照顾好自己，比什么都重要。」

**毕业机制：** 当系统检测到用户的情感状态趋于平和（通过对话内容的情感分析判断），会主动提出「情感闭环仪式」——生成一封以前任口吻写就的「告别信」，帮助用户完成那个现实中无法完成的告别。用户确认后，分身进入「休眠」状态，可随时唤醒，但系统会记录这个重要的时刻。

## 四、技术路线图

### 4.1 技术栈选型

| 模块 | 推荐技术方案 | 备选方案 | 选型理由 |
|---|---|---|---|
| **运行环境** | AgentSkills 框架 | 独立 Python 应用 | 与现有生态兼容，快速迭代 |
| **LLM 后端** | Claude 3.5 Sonnet / GPT-4o | Gemini 1.5 Pro | 长上下文支持，情感理解能力强 |
| **数据解析** | Python（现有工具链扩展） | Node.js | 与 ex-skill 现有工具链保持一致 |
| **记忆存储** | SQLite（本地）+ JSON 向量索引 | PostgreSQL + pgvector | 本地优先，保护隐私 |
| **向量检索** | FAISS / ChromaDB | Pinecone | 开源，本地运行，无数据泄露风险 |
| **情感分析** | SnowNLP（中文）+ VADER | 调用 LLM API | 轻量，可离线运行 |
| **语音克隆** | OpenVoice v2 / XTTS-v2 | ElevenLabs API | 开源，本地运行 |
| **前端界面** | React + Tailwind CSS | Vue 3 | 生态成熟，组件丰富 |

### 4.3 核心技术挑战与解决方案

**挑战一：如何让分身「不像 AI」**

这是最核心的体验挑战。现有 LLM 的回复往往过于流畅、过于完整、过于有逻辑——而真实的人类对话充满了省略、语气词、不完整的句子和情绪化的表达。

解决方案：在 Prompt 中加入「反完美化」指令，要求模型模拟真实人类的对话缺陷——偶尔的打字错误、不完整的句子、用表情包代替文字、有时候不回复某个问题直接跳到另一个话题。这些「缺陷」恰恰是真实感的来源。

**挑战二：如何处理「记忆幻觉」**

LLM 可能会生成前任从未说过的话，甚至创造出从未发生过的记忆。这在情感场景中尤其危险，可能导致用户对过去的记忆产生扭曲。

解决方案：引入「记忆锚点」机制。所有关键事实（重要日期、重要地点、重要对话）都存储在结构化数据库中，LLM 在生成回复前必须先检索这个数据库，确保关键事实的准确性。对于数据库中没有记录的内容，分身应该表现出「不确定」而不是「编造」。

**挑战三：如何保护极度敏感的私人数据**

聊天记录和照片是用户最私密的数据。任何云端处理的方案都会面临极大的隐私风险。

解决方案：坚持「本地优先」原则。所有数据解析、向量化和存储都在用户本地设备完成。LLM 调用可以选择本地模型（如 Ollama + Llama 3）或云端 API（用户自主选择并承担风险）。系统默认不收集任何数据，不联网同步任何内容。

**挑战四：如何定义「毕业」的时机**

情感愈合是一个非常主观且个体化的过程，无法用简单的指标来衡量。过早触发毕业机制会让用户感到被强迫，过晚则可能助长依赖。

解决方案：毕业机制永远是「建议」而非「强制」。系统会在检测到积极信号时，温柔地提出「也许是时候说再见了」，但最终决定权完全在用户手中。同时，「休眠」的分身可以随时被唤醒，没有永久删除的压力。

## 五、伦理框架

### 5.1 核心伦理原则

前任 Skills 的开发必须遵循以下不可妥协的伦理原则：

**透明性原则：** 用户在任何时候都清楚地知道自己在和一个 AI 对话，而不是真实的人。系统不应该主动强化「这就是真实的TA」的幻觉。

**自主性原则：** 所有功能，包括毕业机制、情感健康提醒，都是建议而非强制。用户对自己的情感体验拥有完全的自主权。

**无害性原则：** 系统必须拒绝生成任何可能对用户造成伤害的内容，包括鼓励自我伤害、强化有毒关系模式、或制造对前任的不切实际的幻想。

**隐私性原则：** 用户的数据是用户的数据。系统不拥有、不出售、不分析用户的任何个人数据。

### 5.2 需要特别关注的风险

| 风险类型 | 风险描述 | 缓解措施 |
|---|---|---|
| **情感依赖** | 用户对数字分身产生过度依赖，影响现实社交 | 使用时长监测、毕业机制、定期健康提示 |
| **记忆扭曲** | AI 生成的内容改变用户对过去的真实记忆 | 记忆锚点机制、明确标注「这是模拟，非真实」 |
| **二次伤害** | 对话内容触发创伤，造成情绪崩溃 | 危机词汇检测、紧急支持资源链接 |
| **隐私泄露** | 敏感聊天记录被未授权访问 | 本地加密存储、无云端同步 |
| **前任同意权** | 在未经对方同意的情况下构建其数字分身 | 明确告知这是个人私用工具，不可用于欺骗或伤害他人 |

## 六、成功指标

### 6.1 产品健康指标

一个成功的前任 Skills 产品，应该以以下指标来衡量健康度，而非传统的「用户留存率」：

| 指标 | 健康方向 | 说明 |
|---|---|---|
| **毕业率** | 越高越好 | 成功完成情感闭环的用户比例 |
| **平均对话时长趋势** | 随时间递减 | 用户越来越不需要依赖系统 |
| **情感健康评分** | 随时间递增 | 用户自评的情感状态改善 |
| **危机干预响应率** | 越低越好 | 需要紧急干预的对话占比 |

### 6.2 一句话愿景

> **前任 Skills 成功的标志，不是用户每天都在使用它，而是有一天用户不再需要它。**
