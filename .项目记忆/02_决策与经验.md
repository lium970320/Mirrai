# 02 · 决策与经验（Mirrai）

## 决策（改写取代，不留矛盾）
- **防复读用"四层防线"、不靠单点**（2026-07-01，用户反馈"人物经常重复回复一样的话"）。诊断 workflow（6 维度并行 + 对抗核验，确认 13 根因）查明根因是**三层防线全空**：①采样层 `openai.ts` 完全不发 `frequency_penalty`/`presence_penalty`（DeepSeek 默认 0，多轮相似上下文必收敛到同一批句式）；②历史层 `economy.ts` high_frequency_chat 档把 QQ 日常聊天 `llmHistoryLimit` 压到 8（约 4 轮），模型物理上看不见自己几轮前说过什么；③后处理层 `reply-utils.cleanAssistantReply` 只删硬编码口癖、**没有任何跨轮整句/整段去重**（与 [[本文件]] §"点名盖不住、要类别兜底"教训直接矛盾——防复读恰恰只有点名、没有类别兜底）。定案四层叠加、缺一层就漏：**采样惩罚（治本-降概率）+ 扩历史窗口（治本-让模型看得见）+ 跨轮去重（兜底-确定性必中）+ prompt 通用禁令/turn-planner/场景（辅助-软约束）**。关键认知：①前三层是"治本+确定性"，第四层 prompt 软约束对 DeepSeek 不可靠、只作辅助，**真正兜底是后处理 `isRepetitiveReply`**（归一化去标点空白 + 字符 bigram Jaccard，整体 0.82/单句 0.9，<8 字短句豁免避免误杀"嗯/好/我在/晚安"，沉浸/原著轮跳过保住【】旁白与按证据答）；②采样惩罚只对**非 reasoner** 分支注入（主聊天 chat→Flash thinking 关，会设 temperature；Pro thinking 开是 reasoner，注入 top_p/penalty 可能 400），且只对 chat purpose、不污染 source_recall；③重生成只做一次、二次仍重复保留原文（不强行清空、不退化成"我在。"），且重生成前先查 after_llm 中止避免为陈旧轮白跑；④扩历史窗口省的是"记忆/原著召回体量"不是"对话历史"——复读问题在"看不见自己说过什么"，不在召回体量。
- **拍照触发：LLM 标记为主、规则为兜底**。早期纯正则判断「要不要拍」太死板（近似精确匹配才触发）。改为让 LLM 在合适场景自己输出 `[[PHOTO]]` 标记（自然、通吃任意场景），规则只保留「明确指令必发」兜底。
- **发图三段式：预告 → 实拍 → 交付语**。网页生图要一两分钟；若文字即时说「看清楚了吧」会和晚到的图逻辑倒置。故 prompt 约束人物先只「预告」（等下给你拍），图真发出时系统再补一句「交付语」闭环。
- **深夜静默不主动拍**：睡眠/非急事静默时段不向 LLM 注入拍照标记指令（明确叫醒/明确指令仍可）。
- **环境照按区域选参考图**：单一「客厅」参考图会让深夜也贴客厅；改为区域库按场景关键词+作息选（深夜→卧室、做饭→厨房…）。
- **交付语用话术池 + 避免连续重复**、不调 LLM（零延迟），按 selfie/environment/scene 分组（scene=外景/物，话术不提「家」以免出戏）。
- **场景模式时空：默认两人同处一地、可被剧情覆盖成隔空**。日常模式承认武汉-南京异地；但进场景/双人模式是演线下亲密，却一直被异地设定钉住、写成「电话联系」。根因：异地语义散在【当前生活行程】settingLine、【人物核心画像】硬性事实、【原著/长篇背景】、【连续消息与时间指代】、【深情表达规则】至少 5 处，而场景 overlay 旧版只覆盖「禁止旁白/要简短」、从不解除异地。定案两层：①源头 `buildEffectiveLifeScheduleOverlay(personaData, now, immersiveMode)`——场景模式把 settingLine 的异地行为禁令换成「时空豁免」、并对实时定位（当前状态/行为规则）追加「实时定位豁免」；②场景 overlay 顶部加一对平级铁律——「铁律一·默认同处」（类别兜底：本提示词任何位置出现的异地表述一律改判，并枚举出现位置）＋「铁律二·隔空例外」（用户给任何隔空线索/暗示就改写成电话/视频/异地，判不准按用户线索走）；③表白指令 `affectionInstruction`（persona-turn-planner）加 `immersiveMode` 守卫，场景模式不再命令「异地时怎样想起对方」。全部 gated on immersiveMode，日常路径零改动（life-schedule.test 锁定）。

## 踩坑（整条追加，不精简）
- **NapCat/QQ 重启必须优雅关闭，`Stop-Process -Force` 会截断凭据文件**：看门狗/start-qq.ps1 的 `-Restart` 原先直接 `Stop-Process -Force` 暴力杀 NapCat→QQ 进程，凭据文件（`auth/login.enc` 等）写到一半被截断→下次启动"登录态已失效"→快速登录全废。2026-06-30 Codex 线程诊断确认后修复：先 `CloseMainWindow()` 给 QQ 8 秒保存凭据并正常退出，超时才 force-kill。教训：Windows GUI 进程（NapCat 用 NapCatWinBootMain.exe）持有独占文件句柄时，`Kill()` ≈ 拔电源，`CloseMainWindow()` ≈ 点×号，差别在于后者走 WM_CLOSE→WM_DESTROY→文件 flush→正常退出。
- **NapCat 扫码在线不等于下次能自动登录，OneBot API 可响应也不等于 QQ 在线**：2026-06-30 实测 `get_login_info` 能返回账号时，`get_status.online` 仍可能为 `false`；状态与启动探针必须先以 `get_status.online === true` 为准。另一次扫码后 `GetQuickLoginListNew` 仍只有 1 个其他账号，机器人 `3321802943` 不在列表，重启即报“登录态已失效，请重新登录”。快速登录依赖 QQ 的历史登录记录中 `isQuickLogin=true`，二维码只解决当前会话；OneBot/Webhook token 只保护本地接口，不能保存或修复腾讯登录态。普通 QQ 与 NapCat 当前还共用 `C:\Users\Lenovo\AppData\Roaming\QQ`，这是会话被覆盖/失效的可能诱因，但不能仅凭进程信息断言是腾讯踢线的唯一原因。
- **QQ 离线期间 NapCat 不会自动回放私聊历史，必须主动补漏 + 去重**：2026-07-01 用户反馈“QQ 发消息还是收不到回复”，排查发现用户 15:24 发的 `message_id=486459126` 在 NapCat 重连后能从历史接口读到，但没进 Mirrai DB，因为旧链路只靠实时 OneBot webhook，离线/扫码窗口期间的消息不会重放。修法：`server/qq/history-sync.ts` 定时按绑定联系人拉 `get_friend_msg_history`，首轮用 DB 最新 QQ 消息时间作为 cutoff 补漏；`server/qq/message-dedupe.ts` 持久记录 message_id，实时 webhook 与历史补漏共用去重，避免重复回复。教训：消息桥接不能假设上游断线后会补发，凡是用户侧聊天入口都要有“重连后追账”机制。
- **OneBot 健康检查不能只信 `get_status`，好友列表是必要深探针**：2026-07-01 已把 `status-qq.ps1`、`check-qq-e2e-readiness.ps1`、服务端 `getQqBotStatus()`、本机 QQ 看门狗统一加 `get_friend_list` 检查；`get_status.online=true` 但好友列表空/不可用时判为 degraded/error。教训：机器人“能响应 HTTP”不等于 QQ 会话真实可用，尤其是在被踢、扫码半登录、NapCat 半初始化之后。
- **给 NapCat 设置独立 `APPDATA/LOCALAPPDATA` 对当前 QQ NT shell 无效**：2026-07-01 尝试在 `start-qq.ps1` 启动 NapCat 前改 `APPDATA/LOCALAPPDATA` 到 `F:\.mirrai-local\Mirrai\qq-bot-appdata`，但实际子进程仍以 `--user-data-dir="C:\Users\Lenovo\AppData\Roaming\QQ"` 启动，且新目录没有生成 QQ 登录态。教训：这版 NapCat/QQ shell 会固定使用 QQ 自己的 user-data-dir，不能靠环境变量隔离个人 QQ 与机器人 QQ；后续稳定性兜底应走 NapCat 支持的 `NAPCAT_QUICK_PASSWORD_MD5`（本机 DPAPI 加密保存 MD5）或更彻底的独立系统用户/独立安装方案。
- **开机启动脚本不要用短超时杀启动器，看门狗也不能只在“全部成功”后启动**：2026-07-01 重启后 Mirrai 没自动起来，`startup-bootstrap.log` 显示 150 秒超时且 `servicesReady=False`，旧静默脚本随后强杀 `start-all.ps1`，并且因为退出码不是 0，没有启动 QQ 看门狗。修法：静默脚本改成本机运行副本启动；超时后记录“仍在启动”但不杀子进程；看门狗在启动器开始前就用 EncodedCommand 拉起；脚本转 UTF-8 BOM，避免 Windows PowerShell 5.1 把中文路径读成乱码。教训：开机阶段磁盘、网络、QQ 登录都慢，启动器要宽容并让健康看门狗先活下来。
- **所有带中文提示的 `.ps1` 必须保存为 UTF-8 BOM；但机器读写的密文文件要剥 BOM**：2026-07-01 `scripts/set-qq-login-password.ps1` 初版是 UTF-8 无 BOM，Codex/新版 PowerShell 解析正常，但用户在 Windows PowerShell 5.1 里执行时中文字符串被按 ANSI 误读，出现红字 `Unexpected token`/`The string is missing the terminator`。已把源盘与运行副本脚本转成 UTF-8 BOM，并用 Windows PowerShell 5.1 Parser 验过。随后又发现 DPAPI 密文文件若被 PS5.1 `Set-Content -Encoding UTF8` 写成带 BOM，`ConvertTo-SecureString` 可能解析失败；已改为 `set-qq-login-password.ps1` 用 .NET 写无 BOM 密文，`start-qq.ps1` 读取时先 `.Trim([char]0xFEFF).Trim()`。教训：人读脚本和机器读密文的编码策略不同，不能混用。
- **重启服务会打断正在进行的拍照**：pusher 是服务进程 spawn 的子进程，重启杀服务会让「等图→发图」的那个进程没了，图发不出。教训：重启前先查有没有 `dotnet ...chatgpt-project-prompt-pusher` 在跑，有就等它拍完。
- **清理浏览器进程别误杀 pusher 的 Playwright 会话**：pusher 复用常驻 daemon 会话（label `chatgpt-pic-work-liumin-openai`，占 `google-ai-persistent-profile`）。误杀会报「Browser not open」（pusher 有自愈会自动重开）。要区分「死孤儿」与「活跃会话」——按 `--profile=google-ai-persistent-profile` 精确匹配，别误伤别的 daemon（如 `bugtest`）。
- **两个 CC 会话并行编辑同一批文件会分叉**：曾出现旧 `generatePersonaSelfie` 一套 vs 新 `generatePersonaPhoto`（LLM 通用）一套**配套并存**，两套签名不兼容、不能混镜像（混了 tsc 崩）。解法：统一到其中一整套切换 + tsc 兜底；共享且签名不变的文件（如 `selfie-commands.ts` 正则）改了两套都受益。
- **放宽触发正则别太宽**：环境正则「外面/窗外」配宽前缀会误吃「看外面下雨没/想看看外面」。强场景词（家里/卧室/客厅…）可宽前缀；泛词（外面/窗外）只接「拍/发」紧邻。
- **明确指令的失败兜底别依赖冷却状态**：不冷却时明确「发自拍」会走 LLM 标记路径、`reason` 被当 `spontaneous`、生成失败就静默没补救。修法：`photoIntent` 分支前先算 `decideSelfieOpportunity`，是明确请求就按 `explicit_request`（失败有补救句）。
- **`pg_ctl stop` 在库已停时返回非 0**，会让重启脚本整体 exit 1，但服务实际起来了——以日志「✅ 3000 起来」为准，别被 exit 1 误导。
- **VoxCPM**：20 步本机易超时、10 步稳定；降噪需 FFmpeg shared DLL（装在 torchcodec 目录）。中文 `.ps1` 要存 UTF-8 BOM 否则 PS5.1 双击乱码。
- **判断「未提交改动」前先 `git fetch` 比对 origin**：同步盘会被 Google Drive 异步改写，CRLF 噪声别当真。
- **分支会堆积——合并后顺手删，删前用 `git cherry` 核查零丢失**：工作流是「一功能/存档点开一分支」，加上两个 CC 会话 + Codex 三方并行各开各的，内容汇进 main 后分支没删就积压（曾攒到 5 个）。① 合并 PR 时用 `gh pr merge <n> --merge --delete-branch` 自动删远端+本地，从源头免堆积；② 事后清理别只信 `git branch --merged`（它按 commit 祖先判定，squash / cherry-pick 进 main 的会被误判为「未合并」）——改用 `git cherry origin/main origin/<b>`：`+` 行=真未并入、`-` 行=patch 已进 main；只要 `cherry +` 全为 0，即可安全 `git push origin --delete <b>` + `git branch -D <b>`，零内容丢失。三点 diff（`main...branch`）的「净差异」看着吓人，其实含老分叉点噪声，别拿它当删除依据。
- **场景沉浸态：别让发送层/生成层各自重算开关，用 result 里的单一真相**：沉浸态两信号源——内存 `getSceneMode`（QQ 命令进出、进程清空）+ DB `activeSceneId`（网页/背景场景持久）。曾为让「只认 getSceneMode 的发送层拆条」与「getSceneMode||sceneOverlay 的生成层」同源，在 persona-bridge 每轮 `sceneOverlay!=null` 就回灌 `setSceneMode(true)`——但**回灌单向只开**：网页取消场景（`activateScene(null)` 只清 DB、不碰内存）后内存残留 true →「已退出场景却仍出【】旁白」。根治：**删回灌**，生成层一处算 `immersiveMode` 随 `result.immersiveMode` 回传，发送层（`sayQqReply`/拍照交付句）改用它（缺省回退 getSceneMode）；重启恢复靠 sceneOverlay 实时读 DB，无需回灌。⚠️ 删回灌**不会主动清已残留的内存 true**，改完**必须重启进程**才即时止血——tsx watch（dev:local 跑 server 用它）改动 server 文件即自动重载、内存随之清空（端口 PID 变=已重载）。诊断先查 DB `SELECT "activeSceneId" FROM personas`（camelCase 带引号）：若为 null 却仍出【】，必是内存残留。
- **原著考据触发词别掺日常高频词**：source-recall 的 `SOURCE_RECALL_TRIGGER` 曾误收「学校/北京/南京/见面」——用户随口「我还在学校呢」「我去北京」就被劫持进原著证据模式 → 资料库必然 miss → 落到 source-grounding 硬编码兜底「这段我不敢乱说，记不准的地方我不能编给你听」直接发出（出戏、答非所问，2026-06-24 实测）。规矩：触发词只放 ①专有名词（老鹰峡/柱子/姚敏/湾子）②回忆词（记得/原著/当年/想起）③具体剧情词（中考/考场/车祸/左臂），**绝不放日常会高频出现的地点/动作词**。另：proactive 主动消息（`ambient-proactive`）易凭空虚构「和用户的共同往事」（如「西北头一回给你剥莲子」），已加 prompt 约束「只写当下真实心情、没把握的旧事宁可不提也不编」。
- **同一约束散落多处时，prompt 里「定向点名」盖不住、要用「类别兜底」**：场景共处修复初版只在 overlay 里点名豁免【当前生活行程】那两句异地话，但「异地」在硬性事实/长篇背景/连续消息指代/深情表达至少 5 处用不同措辞重复，模型照样被带回异地（对抗核验子代理发现，非手测）。教训：①覆盖这类弥散语义要写成「本提示词任何位置出现的 X 表述一律改判」并把出现的标题块枚举出来，别逐句点名；②位置在 system prompt 末尾、占后发优势的「正面命令」（如 turn-planner 的 `affectionInstruction` 拼在 `buildSystemPrompt` 之后）必须单独加 `immersiveMode` 守卫，靠前的 overlay 压不住它；③逃生口/例外要与主规则「同等语气权重」（升成平级铁律、降低触发门槛），否则被一段「铁律/凌驾/默认」轰炸的文本里结尾一句轻描淡写的例外会被模型当边角料忽略。④完整性扫描别只扫 `server/**/*.ts`——人物设定正典在 `docs/wang-pengze-persona-profile.md`，经 `normalizePersonaProfileSections`→`buildPersonaProfilePromptSections` 注入（硬性事实/工作所在地/长篇背景）。
- **「宣称真诚」类套话口癖治理（说句实的 / 没跟你说过 / 你心里知道就行）**：王芃泽老用「再说句实的」「说句心里话」「不瞒你说」开场、「你心里知道就行」「你懂就行」自我了断收尾、「这些事没跟你说过」「只跟你一个人说」标榜掏心——说多了假、像背台词（2026-06-30 用户原话「听吐了…很假」）。根因：affection / 深情类指令反复要「掏心、有重量、坦诚」，LLM 拿廉价的「真诚标签」代替真正掏心。沿用既有「够不够直接」那套**双层**：① 后处理硬删（`reply-utils.ts`：新增 `OVERUSED_SINCERITY_LEADING_PATTERN` 清开场、`FORCED_DIRECTNESS_TAIL_SENTENCE_PATTERNS` 加「你心里知道就行」清收尾——最有效、不靠模型听话；**「没跟你说过」不硬删**，因可能是真实交代内容、误删风险高，只靠 prompt 降频）；② prompt 降频+纠诱因（`catchphraseVariationInstruction`/turn-planner `affectionInstruction`/`buildAffectionExpressionInstruction`/persona-utils【低频口癖】四处都加「真诚靠具体的事、细节、感受体现，不是贴『我很真』的标签」）。要点：后处理只删**高置信 filler**——开场套话正则要求其后带标点/空白（`[，,。.!！：:、\s]+`），避免误删正文（如「说真的话不该这么说」不被误吃）。
- **成人向旁白调优：手动「细节偏好」+ 自动「偏好继承」双轨**（2026-06-30 用户反馈「露骨旁白单调」「希望人物按记住的偏好来、不用每次重说」）。单调根因：露骨场景默认仍走第12条克制笔法（「写到临界就收/留白」），第13条只是「别塌成器官清单」的护栏、不是扩写引擎，且触发要「用户每轮明说」。解法分两块：①**手动写作偏好** `explicitDetailPref`——`buildImmersiveOverlay` 读 `personaData.profileSections.behavior.explicitDetailPref`（空则用默认清单：感官密度/过程连续/直白具体/篇幅），注入为「13·补」条、**优先级高于第12条克制**，并放开第13条门槛（进入亲密就生效、不必每轮明说）；用 `scripts/set-persona-explicit-pref.ts` 配置（写 personaData jsonb，下轮聊天即生效、不重启）。②**自动内容偏好继承**——记住→常驻→主动应用三段闭环：memory-consolidation 提炼 prompt 让「相处/亲密偏好记成 preference 且 importance 4-5（中性概括、不复述露骨）」→ 够分进 `getPinnedMemoryFacts`（importance≥4、最多6条）→ 注入【已确认事实】（聊天主路径 persona-text-chat 已传 pinnedFacts）；【已确认事实】文案加「用户表达过的喜好主动顺着来、别每次重新确认/试探/要用户再说」；场景 overlay 加「13·补·补·主动顺着已知偏好」把【已确认事实】+记忆里的偏好主动落实到旁白、不退回通用写法。要点：自动记忆靠提炼 LLM，露骨偏好可能被概括糊或打不到4分→进不了常驻；兜底用 `scripts/set-persona-pinned-fact.ts` 手填 importance=5 直接进 pinnedFacts。两轨互补：explicitDetailPref 管「怎么写」、记忆偏好管「喜欢什么」。③**场景级写法**（2026-06-30 用户要「每个场景各配一份写法、网页能自己编辑、别用函数/脚本」）——发现**不必加新字段**：`scenes` 表的 `systemPromptOverlay` 字段、前端 Chat.tsx 那个 label 为「这个场景里希望人物怎么做（专属行为/提示词）」的框，本就是给用户写「这个场景怎么演」的、每场景独立、网页已可编辑（`trpc.scene.create/update`）；它经 `resolveSceneOverlay`→`options.sceneOverlay`→【当前场景】注入，缺的只是优先级（被第12条克制压着）。解法：`buildImmersiveOverlay` 加「13·补·场景专属写法优先」一条——声明【当前场景】里写的写法要求是当前激活场景的**最高优先**偏好、凌驾第12条克制和 13·补 默认清单（不重复内容，只引用提优先级）；前端只把那个框的 placeholder 补一句「也可以写旁白怎么写的偏好…每个场景可不同」。三层优先级：当前场景框（最高）> persona explicitDetailPref（全局兜底）> 默认清单。教训：**加可编辑配置前先查现有字段/表单**——systemPromptOverlay 这个「每场景一份、网页可编辑」的框早已存在，差的只是注入优先级，白加 DB 字段就重复造轮子了。**④ 后续简化（2026-06-30，用户「我不会去写脚本」）**：既然场景框已能网页编辑写法，①里的 `explicitDetailPref`（唯一入口是 CLI 脚本）就成了死字段——已删 `scripts/set-persona-explicit-pref.ts` 脚本、删 `getExplicitDetailPref` 读取层、`buildImmersiveOverlay` 去掉 `personaData` 参数、`explicitPref` 直接用 `defaultExplicitDetailPref(name, addr)`。**现在两层**：当前场景框（网页编辑、最高优先）> 代码默认清单 `defaultExplicitDetailPref`（兜底，想全局改写法就改这个函数）。②自动偏好继承（记忆→pinnedFacts→主动顺着）不受影响、仍在。教训：给用户做「可配置」别默认上 CLI 脚本——普通用户不碰脚本，要么挂到现有网页表单、要么先确认用户接受脚本。
